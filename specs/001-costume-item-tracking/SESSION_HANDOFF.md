## 2026-07-12 素材位置/尺寸大修（新對話必讀）

上一輪只修好魔杖，使用者回報「幾乎所有素材位置/大小比例都跟設計圖有落差，只有
卷軸、魔杖、寶箱基本正確」。這輪用 Python 對設計參考圖
(`C:\Users\USER\Downloads\ab72061a-bd50-4eab-b446-07e58d3ca81b.jpg`, 1024×755) 疊網格線
精確量測像素座標，逐一比對，發現兩個根本原因：

1. **COSER 素材本身四周留白不對稱**：`coser.png` 畫布 1824×1643，實際圖案只佔
   canvas 下方 74% 高度（上面 26% 是透明留白）。用 `object-fit:contain` 時，這段
   留白會把可見的貓耳臉孔往下推，導致臉孔看起來偏低、卡在輸入框上而不是貼齊卡片頂端。
   **修法：用負的 `top` 把留白裁掉**（卡片本身 `overflow-hidden` 會吃掉裁切，可見內容
   完全不受影響），而不是只調整可見框的 width/height。已修正：
   `top:-2rem;right:2px;width:175px;height:158px`（原本 `top:0.5rem;right:0.5rem;
   width:185px;height:167px`，症狀是位置對但视觉偏低、偏小）。
2. **設計參考圖本身跟實際頁面的版面比例不一致**（mockup 是獨立 1024px 寬的靜態視覺稿，
   header/卡片間距跟我們實際 Tailwind build 出來的頁面不是線性縮放關係），純粹套「量到的
   px ÷ 參考圖卡片寬 × 實際卡片寬」公式换算，遇到頁面級背景出血素材（long-wig/
   short-wig/sword/costume）會算出離譜甚至爆版寬捲軸的數字。**改用「先精確量卡片內
   素材（有明確卡片邊界可鎖定）、頁面出血素材用視覺比對＋保守微調」的混合法**，每次
   改完都用 Playwright 截圖後跟參考圖同寬縮放疊圖比對，而不是隻信數學公式。

這輪實際調整（`public/index.html` 的 `<img>` inline style）：

  「角色完整清單」卡片太多；參考圖裡它應該小小一個卡在兩張卡片中間的窄縫）
- `wand.png` / `chest.png` / `scroll.png` / `logo.png`：**沒有動**（使用者確認這三個

## 2026-07-12 第二輪：疊網格量測還是不夠準，改用「實際素材疊圖比對法」

使用者回報上一輪修完後 COSER/長假髮/劍/服裝 這 4 個還是位置+大小都不對（劍還被抓到
「太偏左」）。深入排查後發現：**「疊網格線量測參考圖、算比例、套公式換算成即時頁面
px」這個方法本身有兩個結構性誤差來源，導致永遠對不準**：

1. **參考圖是 1024px 寬的靜態設計稿，跟我們用 Tailwind build 出來的實際頁面版面比例
   不是線性縮放關係**（mockup 的 header 高度、卡片間距是設計師手調的，不是跟卡片寬度
   同比例縮放）。垂直方向套用「卡片寬度比例」公式換算會系統性算錯（曾經算出要橫向
   溢出 200px+ 的離譜數字）。
2. **肉眼在模糊的 JPG 上讀網格線座標，精度撐死到 ±20-30px**（在一張 423px 寬的卡片
   範圍內，這已經是很大的誤差比例了），尤其線稿類素材本身在參考圖上就已經是很淡的
   灰色描邊，跟背景對比度低，容易讀錯轉折點/量到相鄰但不同的線條（劍那次就量到錯的
   線，量出來反而叫它更往左移，跟使用者說的方向完全相反）。

**改用的正確方法：把「實際要用的那個透明背景 PNG」直接疊在參考圖上、套紅色調方便辨識，
反覆試位置/大小、每次存檔用 Read 工具實際看疊合結果，直到紅色描邊跟參考圖底下那條灰線
對齊為止**——不再是「量座標→算公式→套用」，而是「試疊圖→眼睛看有沒有對齊→調整→
再試」的直接視覺收斂迴圈。準確度高很多，因為比較的是同一張參考圖上「素材本尊疊上去對
不對齊」，不會有 mockup 版面比例跟即時頁面不一致的換算誤差。COSER 就是用這個方法連續
試了 5 次疊圖才收斂到位置（見下方最終數值），劍也是重新試了 2 次才發現第一次疊圖疊到
錯的參考線上。

這輪最終套用的數值（`public/index.html`）：
- `coser.png`：`top:0;right:17px;width:172px;height:155px`（跟卡片內距對齊，右邊緣切齊
  輸入框右緣，不是切齊卡片邊緣——這是使用者這輪指出的關鍵線索）
- `long-wig.png`：`top:24px;right:-56px;width:384px;height:384px`
- `short-wig.png`：`top:442px;left:-22px;width:252px;height:252px`
- `sword.png`：`top:418px;left:218px;width:312px;height:312px`（尺寸加大是關鍵，劍柄
  是視覺重心，框太小會讓劍柄看起來偏左偏下，不只是位移問題）
- `costume.png`：`top:536px;right:-64px;width:228px;height:228px`（原本太小且太靠下）

跑過 1280/1024 兩個寬度確認 `scrollWidth <= clientWidth`（無橫向捲軸），390×900/768×1000
也截圖確認沒跑版。

**如果下一輪還要繼續微調**：直接用這輪的「疊圖試位置」方法（`.dev-tools/` 底下寫一個
臨時 Python script，把 `public/assets/<file>.png` 用 PIL resize+紅色調疊到參考圖
`C:\Users\USER\Downloads\ab72061a-bd50-4eab-b446-07e58d3ca81b.jpg` 上、存檔、用 Read
工具看、調整參數重試），不要回頭用網格線量座標套公式的方法——那個方法這兩輪都被證明
不夠準。試出滿意的 ref-space 座標後，再乘上卡片寬度比例（`live卡片寬508 ÷ ref卡片寬423
≈ 1.2009`）換算成即時頁面的 px，套進 `index.html`。

## 2026-07-12 第三輪：疊圖法對長假髮也失敗，改用「即時頁面 DOM 幾何」定位法（目前有效）

第二輪的「拿實際素材疊圖比對參考圖」方法，對 COSER 有效，但對長假髮這種本身是「手繪
線稿」風格、跟參考圖裡的手繪草稿長相本來就不完全一樣的素材，疊圖比對會失真——
會疊出一個「疊起來看起來像」但實際尺寸判斷錯誤的結果，這輪把長假髮越修越大，跟
使用者要的方向完全相反。**結論：疊圖法不是萬用的，遇到「参考圖是手繪草稿、實際素材
是另一份完稿」這種情況會不準。**

**這輪改用的方法，目前驗證有效**：不再嘗試跟參考圖比對，改成直接用 Playwright 量出
「即時頁面上真實 DOM 元素」的精確座標（輸入框、卡片邊界，這些是 100% 精確的數字，
不是量測猜的），再套用使用者給的具體方向性描述（例如「右邊與輸入框齊平」「劍尖被
搜尋道具卡片蓋住一點」「一半範圍壓在兩張卡片下」）直接算出目標 px，而不是再去比對
參考圖。這個方法的優點：所有座標都是精確量出來的（`page.$(sel).boundingBox()`），
不會有 JPG 模糊/mockup 比例不一致的誤差來源；缺點：需要使用者給明確的相對描述（跟哪個
元素切齊、蓋住多少），不能單靠參考圖自動算。

這輪最終數值（`public/index.html`）：
- `coser.png`：`top:8px;right:25px;width:117px;height:105px`（右邊=輸入框右緣、
  底部=輸入框頂緣，算法：`input.boundingBox()` 減去 `card.boundingBox()` 座標差）
- `long-wig.png`：`width/height 200×200`（從上一輪的 384 大幅縮小回去，位置
  `top:10px;right:-56px` 不變，只讓左下角壓進卡片一點點）
- `sword.png`：`top:464px`（比上一輪下移 46px，讓劍尖精確蓋進「搜尋道具」卡片
  `top:740.8px` 一點點，已截圖確認蓋住效果正確）
- `costume.png`：`top:576px;right:-89px`（往下+往右移，降低跟兩張卡片的重疊面積到
  接近一半）

1280/1024 寬度確認過沒有橫向捲軸。

**下一輪如果還要調**：優先用這輪的「量 DOM 座標 + 套使用者的相對描述」方法，不要
回去用疊圖法或網格量測參考圖法——這兩個方法都已經被驗證對這個案子不夠準/會出錯。
如果使用者這輪回饋還有不滿意的地方，請他們用「跟哪個元素切齊/對齊」「蓋住多少」
這種相對於「頁面上其他元素」的具體描述（不要只講「太大/太小/位置不對」），這樣才能
用精確 DOM 座標算，不用再猜。

## 這個 session 做了什麼

把 `costume-manager`（單機服裝道具管理工具）的前端從純手刻 CSS 改成：
**Tailwind CSS v4 + DaisyUI v5（本地建置，非 CDN）**，套用「韓系插畫風」設計，並把使用者
提供的 9 張素材圖（COSER、劍、長/短假髮、魔杖、卷軸、寶箱、服裝、LOGO）套進對應卡片位置。

同時裝了 **Playwright**（`.dev-tools/shot.mjs`）當開發用截圖工具，因為內建瀏覽器截圖工具
這個 session 大部分時間都連不上——之前很長一段時間是靠算 DOM 座標數字猜版面對不對，
效率很差、常常猜錯。改用 Playwright 真的截圖來看之後，修正效率高很多。
